# 要件定義

## 参考情報
* 高等学校（情報）教員資格認定試験 科目Ⅰの過去問分析記事を参考にする
    * https://qiita.com/misato729/items/c9e35ab0141ab87932dc
* 科目Ⅰは全20問の4択問題で、6割以上が合格ライン
* 出題傾向は以下の大分類で整理する
    * 問1〜問15相当: 教職教養
    * 問16〜問20相当: 情報科特有の問題
* 過去問が少なく傾向が変わる可能性があるため、問題分類や出題範囲はコード上の固定値として後から追加・変更できる設計にする
    * 分野の追加・変更頻度は低い想定のため、管理画面ではなく `utils` フォルダ内の配列で管理する

# 機能要件

## ログイン機能
* 会員登録ができる
    * ユーザー名、メールアドレス、パスワードを登録する
    * メールアドレスは一意とする
    * パスワードは8文字以上とする
* ログインができる
* ログアウトができる
* パスワード再設定ができる
* アカウント削除（物理削除のみ）ができる
    * アカウント削除時、演習履歴・お気に入りも削除する

## 問題閲覧（基本情報技術者試験ドットコムでいう「過去問題解説」）
* 模擬問題、選択肢、正答、解答解説を閲覧することができる
* 問題・選択肢・解答解説は生成AIで作成する
* 問題には以下の情報を持たせる
    * 問題文
    * 4つの選択肢
    * 正答
    * 解答解説
    * 大分類（教職教養、情報科特有）
    * 小分類
        * 教職教養：教育史、教育法規、学習指導要領、生徒指導提要、教育心理、令和の日本型学校教育
        * 情報科特有：学習指導要領(情報)、アルゴリズム、データサイエンス
    * 難易度（★、★★、★★★）
    * 参考にした出題範囲・根拠資料
    * 公開状態（下書き、公開、非公開）
* 誤りを見つけたユーザーが問題の誤りを報告できる
* 管理者は報告内容を確認し、問題・選択肢・解説を修正できる

## 問題演習（基本情報技術者試験ドットコムでいう「過去問道場」）
* 前提としてログインユーザー限定の機能である
* 問題をお気に入り登録することができる
* 問題の正解・不正解を保存できる
* 分野指定で出題することができる
* 以下の条件で出題できる
    * 全分野から出題
    * 大分類を指定して出題
    * 小分類を指定して出題
* 追加条件として以下の条件も設定できる
    * お気に入り問題のみ出題
    * 間違えた問題のみ出題
    * 未回答の問題のみ出題
    * 問題数を指定して出題
* 問題は1回の模擬問題20問を60回分（合計1200問）用意する。
* 60回分の模擬問題セットを20セットずつ3段階に分け、各セットの全問題に★、★★、★★★と振り分ける
* 「模擬試験」形式機能では1回20問のセットで本番同様の分野の問題数（教職教養15問、情報科特有5問）で演習できる
    * 教職教養15問の内訳：教育史2問、教育法規3問、学習指導要領5問、生徒指導提要1問、教育心理3問、令和の日本型学校教育1問
    * 情報科特有5問の内訳：学習指導要領(情報)3問、アルゴリズム1問、データサイエンス1問
    * 1回の演習終了時に、正答数、正答率、合格ライン到達有無を確認できる
    * 合格ラインは12問以上正解とする

## 問題演習履歴閲覧
* 前提としてログインユーザー限定の機能である
* 問題演習機能で解いた問題の進捗状況を確認できる
* 問題ごとに正解回数、不正解回数、最終回答日、連続正解数を確認できる
* メダル表示は以下の条件とする
    * 1回正解すると銅メダル
    * 2回連続正解すると銀メダル
    * 3回連続正解すると金メダル
* 不正解の場合、その問題の連続正解数は0に戻る
* 演習回ごとに、実施日、出題分野、出題数、正答数、正答率を確認できる

## 試験概要ページ閲覧
* 高等学校（情報）教員資格認定試験についての情報を掲載する。
* 1ページ内に試験概要と過去問ベースの出題範囲の解説を掲載する
* 科目Ⅰの概要として、問題数、選択肢形式、合格ライン、出題範囲を掲載する
* 出題範囲は以下の観点で掲載する
    * 教職教養
    * 情報科特有の問題
    * 学習指導要領・学習指導要領解説
    * 教育法規
    * 生徒指導・教育心理
    * アルゴリズム・データサイエンス
* 出題傾向は年度によって変わる可能性があることを明記する

## 管理機能
* 管理者は問題を作成・編集・削除できる
* 管理者は問題の公開状態を変更できる
* 管理者はユーザーからの誤り報告を確認できる
* 管理者は誤り報告への対応状態を管理できる
    * 未対応
    * 対応中
    * 修正済み
    * 対応しない

# 非機能要件
* パスワードはハッシュ化して保存
* 認証方式は Rails API と Nuxt の構成に合わせて決定する
    * Cookie ベース認証またはトークンベース認証を検討する
    * 本番環境では HTTPS を前提とする
* フロントエンドとバックエンドのドメインが分かれるため、CORS 設定を行う
* 個人開発なのでレスポンスが遅すぎなければOK
    * Vercel(FE),Fly.io(BE),Neon(DB)の無料枠で十分な想定
    * 無料枠のスリープにより初回アクセスが遅くなる可能性は許容する
* 経費削減のためドメイン取得は行わない（Vercel上で動かす想定）
* スマートフォンでも主要機能を利用できるレスポンシブデザインとする
* データベースは定期的にバックアップできる構成にする
* 生成AIで作成した問題には誤りが含まれる可能性があるため、ユーザーにその旨を明示する
